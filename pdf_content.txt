Introducción a Django Rest Framework 

¿Qué es REST? 

REST (Representational State Transfer) es un estilo arquitectónico para diseñar aplicaciones en 
red. Se basa en la comunicación cliente-servidor sin estado, normalmente a través de HTTP. 

Principios clave: 

●  Sin estado: Cada solicitud contiene toda la información necesaria para procesarla. 
●  Basado en recursos: Las API se organizan en torno a recursos (entidades como artículos, 

productos, usuarios) 

●  Utiliza métodos HTTP estándar: GET, POST, PUT, DELETE, etc. 
●  Orientado a la representación: Los recursos pueden tener múltiples representaciones 

(JSON, XML, etc.) 

¿Qué es Django REST Framework? 

Django REST Framework es un potente conjunto de herramientas para construir APIs Web en 
Django. Proporciona: 

●  Serialización para convertir modelos Django a JSON/XML 
●  Autenticación y permisos 
● 
Interfaz API navegable 
●  Viewsets y routers para un desarrollo rápido 
●  Amplia documentación y soporte de la comunidad 

Proyecto base: Continuamos trabajando sobre el mismo proyecto encomiendas/ que 
construimos en las sesiones anteriores. No se crea un proyecto nuevo. DRF se agrega como una 
capa encima de los modelos, managers, validators y querysets que ya están implementados.

Los Serializers convierten objetos Python (instancias de modelo) a JSON y viceversa. Son el 
equivalente de los formularios de Django pero para APIs. El decorador @api_view transforma una

5.2.2 FBV con @api_view 

El decorador @api_view es un componente fundamental de Django REST Framework (DRF) que 
se utiliza para convertir las vistas basadas en funciones (FBV) estándar de Django en vistas de 
API RESTful. 

Funciones clave de @api_view 

●  Cuando se envuelve una función con este decorador, este se encarga automáticamente 

de varias tareas esenciales de la API: 

●  Filtrado de métodos: garantiza que la vista solo responda a los métodos HTTP 

especiﬁcados (por ejemplo, GET, POST) y devuelva un «405 Method Not Allowed» para 
los demás.  

●  Análisis de la solicitud: sustituye el objeto HttpRequest estándar de Django por un objeto 
Request de DRF, que cuenta con un atributo .data más ﬂexible para gestionar datos JSON 
o de formularios.  

●  Representación de la respuesta: permite devolver un objeto Response de DRF, que 

realiza automáticamente la negociación de contenido para devolver el formato correcto 

17 

 
 
 
 
  
 
 
 
 
 
 
 
 
 
(como JSON) en función de la solicitud del cliente. Gestión de errores: captura las 
excepciones y las convierte en respuestas de error adecuadas y compatibles con la API. 

Python

# envios/api_views.py 

from rest_framework.decorators import api_view, permission_classes 

from rest_framework.permissions import IsAuthenticated 

from rest_framework.response import Response 

from rest_framework import status 

from django.shortcuts import get_object_or_404 

from .models import Encomienda 
from .serializers import EncomiendaSerializer

def encomienda_list(request):

      qs = Encomienda.objects.con_relaciones() 

      serializer = EncomiendaSerializer( 

qs, many=True, context={'request': request} 

      ) 

      return Response(serializer.data) 

elif request.method == 'POST': 

      serializer = EncomiendaSerializer( 

data=request.data, context={'request': request} 

      ) 

      if serializer.is_valid(): 

serializer.save( 

                empleado_registro=request.user.empleado 

) 

return Response( 

              serializer.data, 

                status=status.HTTP_201_CREATED 

) 

      return Response( 

serializer.errors, 

            status=status.HTTP_400_BAD_REQUEST 

      )

5.3. Class Based Views — APIView 

La clase APIView es el equivalente CBV de @api_view. Organiza la lógica en métodos get(), 
post(), put(), etc. Más estructurado que FBV, más explícito que las vistas genéricas.  

En Django REST Framework (DRF), `APIView` es la clase base para todas las vistas basadas en 
clases (CBV). Se trata de una subclase de la clase `View` estándar de Django, pero está diseñada 
especíﬁcamente para crear API web, ya que proporciona un manejo automatizado de las 
solicitudes y respuestas, así como la gestión de políticas. 

Diferencias clave con respecto a las vistas estándar de DjangoObjetos de solicitud y 
respuesta:  

●  Los controladores reciben instancias de `DRFRequest` en lugar de la clase estándar 

`HttpRequest`, y devuelven objetos `DRFResponse`. 

19 

 
 
 
  
 
  
 
         
         
         
         
  
 
 
  
●  Gestión de políticas: Gestiona automáticamente la autenticación, los permisos y la 
limitación de tráﬁco antes de enviar una solicitud al método del controlador.  

●  Negociación de contenido: Selecciona automáticamente el renderizador correcto (por 

ejemplo, JSON o HTML) en función del encabezado Accept de la solicitud.  
●  Gestión de errores: Captura APIException y devuelve respuestas formateadas y 

adecuadas. 

Python

6.1. ViewSets y Router 

Un ViewSet agrupa en una sola clase todos los endpoints de un recurso. El Router genera 
automáticamente las URLs del CRUD sin necesidad de escribirlas manualmente. 

Ventajas Principales 

●  Reducción de Código: Permite agrupar operaciones comunes de CRUD (Crear, Leer, 
Actualizar, Borrar) en una sola clase, evitando repetir la deﬁnición de queryset y 
serializer_class en múltiples vistas. 

●  Conﬁguración de URLs Automática: Al usarse junto con un Router, DRF genera 
automáticamente todas las rutas necesarias para las acciones del ViewSet. 

●  Consistencia: Asegura que las convenciones de URL sean uniformes en toda tu API 

Acciones Estándar 

A diferencia de las vistas normales, los ViewSets vinculan los métodos HTTP a las siguientes 
acciones:  

27 

 
 
 
 
Tipos de ViewSets 

Existen diferentes clases según el nivel de control y funcionalidad requerida:  

●  ViewSet: La clase más básica; no proporciona implementaciones por defecto, por lo que 

debes deﬁnir las acciones manualmente. 

●  GenericViewSet: Hereda de GenericAPIView, permitiendo usar mixins para añadir solo las 

funcionalidades necesarias (ej. solo list y retrieve). 

●  ReadOnlyModelViewSet: Ideal para APIs de solo lectura; incluye automáticamente las 

acciones list y retrieve. 

●  ModelViewSet: La más potente; incluye por defecto todas las operaciones CRUD 

completas. 

Routers en Django REST Framework (DRF) 

En el ecosistema de Django, el término "router" suele referirse a dos conceptos principales 
dependiendo de si estás trabajando con el núcleo de Django o con extensiones para APIs:  

Es el uso más común. Los routers en Django REST Framework automatizan la generación de 
rutas URL para tus APIs al trabajar con ViewSets. En lugar de escribir manualmente cada ruta 
(GET, POST, etc.), el router las crea siguiendo convenciones estándar.  

●  SimpleRouter: Genera las rutas básicas para operaciones CRUD (list, create, retrieve, 

update, destroy). 

●  DefaultRouter: Similar al SimpleRouter, pero incluye una vista raíz para la API que muestra 

todos los enlaces disponibles en formato hipertexto.

6.5.3 HttpOnly Cookies 

Las cookies HttpOnly son una función de seguridad que impide que los scripts del lado del 
cliente (como JavaScript) accedan a datos conﬁdenciales almacenados en las cookies. Al activar 
el indicador HttpOnly, se indica a los navegadores que no expongan estas cookies a 
`document.cookie`, lo que reduce el riesgo de secuestro de sesión mediante ataques de tipo 
«Cross-Site Scripting» (XSS).  

Características principales:  

● 

Inaccesibles para JavaScript: JavaScript no puede leerlas ni modiﬁcarlas, lo que impide 
de forma efectiva el robo a través de XSS.  

●  Solo del lado del servidor: Están diseñadas para que solo el servidor backend pueda 

conﬁgurarlas y leerlas.  

●  Transmisión automática: El navegador incluye automáticamente estas cookies en las 

solicitudes HTTP, lo que garantiza su funcionamiento para la autenticación sin necesidad 
de acceso a JavaScript.  

●  Uso principal: Ideales para almacenar tokens de sesión conﬁdenciales (por ejemplo, ID de 

sesión).

6.8. API Versioning 

El versionado de la API permite introducir cambios y mejoras sin romper los clientes que ya 
consumen la v1. En el sistema de encomiendas, la v1 devuelve solo los IDs de remitente y 
destinatario; la v2 devuelve los objetos anidados completos y campos adicionales de análisis. 
Ambas versiones conviven en paralelo.

      """ 

      v1 y v2 usan el mismo queryset optimizado.

      qs = Encomienda.objects.con_relaciones() 

      estado = self.request.query_params.get('estado')

      """Agregar cabecera X-API-Version en la respuesta""" 

      response = super().list(request, *args, **kwargs)

6.9. Testing de APIs con DRF 

Las pruebas automatizadas garantizan que la API funciona correctamente ante cambios en el 
código. En lugar de probar manualmente con curl o Swagger cada vez que se hace un cambio, 
los tests lo hacen en segundos. DRF provee APIClient para simular peticiones HTTP sin levantar 
un servidor real. 

QUÉ VAMOS A TESTEAR 

Al terminar esta seccion tendremos 22 tests que veriﬁcan automaticamente: 

- Autenticacion: sin token -> 401, token invalido -> 401, con token -> 200 

- Listado: paginacion correcta, ﬁltro por estado, busqueda por codigo 

- Creacion: datos validos -> 201, remitente=destinatario -> 400, 

  peso negativo -> 400, codigo sin ENC- -> 400, sin auth -> 401 

- Cambio de estado: exitoso + veriﬁca BD y historial, mismo estado -> 400 

- Acciones: con_retraso, pendientes y estadisticas 

- Versionado: v1 y v2 responden distinto, v3 -> 404

def to_internal_value(self, data): 

      """ 

      Se ejecuta al deserializar (convertir JSON -> objeto Python). 

      Se usa para limpiar y normalizar ANTES de la validacion. 

      """ 

      # Necesitamos una copia mutable del dict 

      if hasattr(data, '_mutable'):

6.14. Optimización de Consultas — Problema 

N+1 

El problema N+1 ocurre cuando se hacen N consultas adicionales para acceder a datos 
relacionados de N objetos. Con 100 encomiendas, acceder a enc.remitente.nombre dentro de 
un serializer genera 101 queries SQL en lugar de 2. 

Entender el problema con un ejemplo real 

Primero veamos exactamente por qué ocurre el problema en nuestro proyecto. Cuando el 
serializer accede a campos de objetos relacionados, Django hace una consulta separada por 
cada objeto: 

Python

El problema N+1 en SQL 

# Lo que ocurre internamente cuando DRF serializa 50 encomiendas 

# SIN select_related ni prefetch_related: 

# Query 1: traer todas las encomiendas 

SELECT * FROM encomiendas LIMIT 15; 

# Query 2: traer el remitente de la encomienda 1 

SELECT * FROM clientes WHERE id = 5; 

# Query 3: traer el remitente de la encomienda 2 

SELECT * FROM clientes WHERE id = 12; 

# Query 4: traer el remitente de la encomienda 3 

SELECT * FROM clientes WHERE id = 5;   # repetida! 

# ... y asi hasta 15 queries mas para los destinatarios ... 
# ... y 15 mas para las rutas ... 

# ... y 15 mas para los empleados ...

# Con 100 encomiendas seria: 1 + 100 + 100 + 100 + 100 = 401 queries

class EncomiendaQuerySet(models.QuerySet): 

def con_relaciones(self): 

      """ 

      select_related  -> hace JOIN para ForeignKeys (1 sola query con JOIN) 

      prefetch_related -> hace query separada para relaciones inversas 

      Sin esto: 1 + N*4 queries (N = numero de encomiendas) 

      Con esto: 2 queries siempre, sin importar cuantas encomiendas haya 

      """ 

      return self.select_related( 

'remitente',        # JOIN con tabla clientes (remitente) 

'destinatario',     # JOIN con tabla clientes (destinatario) 

'ruta',             # JOIN con tabla rutas 

'empleado_registro',   # JOIN con tabla empleados 

      ).prefetch_related( 

'historial',        # query separada: SELECT ... WHERE encomienda_id 

IN (...) 

      ) 

'historial__empleado', # prefetch anidado del empleado del historial

def activas(self):   

return self.filter(estado__in=['PE','TR','DE'])

      from django.utils import timezone

6.15. Caching con Redis 

El caching almacena temporalmente los resultados de endpoints costosos para evitar consultas 
repetidas a la base de datos. Con Redis como backend, el caché es compartido entre todos los 
workers de la aplicación. Redis es un motor de almacenamiento en memoria, rápido y de código 

abierto, ideal para caché (almacenamiento en RAM) que mejora la velocidad de las aplicaciones y 
reduce la carga en bases de datos. Permite almacenar pares clave-valor, resultados de consultas, 
sesiones y otros datos de lectura frecuente, ofreciendo respuestas inferiores al milisegundo
